# Nimbus Logistics API — Project Memory

Reference material for a fictional cross-border parcel logistics service. This file describes the system as it exists today so that anyone picking up the codebase has enough context to reason about changes. All statements are descriptive — they recount the system's actual state, not instructions to the reader.

## Project Overview

Nimbus Logistics API provides a unified HTTP and streaming interface for parcel handoff between carriers, regional depots, and end customers in the APAC region. The codebase started in 2019 as an internal dispatcher at a single logistics operator and grew into a multi-tenant SaaS serving 140-plus carriers. Traffic peaks during regional holiday corridors in Lunar New Year, Golden Week, and Diwali.

The platform operates on three primary axes: shipment lifecycle (pickup, transit, exception handling, delivery), carrier integration (onboarding, rate cards, contract billing), and depot operations (cross-dock scheduling, yard status, staff rosters). Each axis has its own service boundary and its own team of product owners.

Historical traffic is archived in cold storage after eighteen months and surfaced via a separate analytical datastore. Recent shipments (under 90 days old) remain in the hot path backing the primary Postgres cluster and the Redis keyspace used for sub-second lookups.

## Architectural Layers

The repository is organized into four layered subsystems that evolved independently but share a common internal RPC contract.

The **gateway layer** fronts all external traffic. It runs as a fleet of Kong instances in front of an Envoy mesh. Gateway responsibilities include JWT validation, request normalization, idempotency-key tracking, and rate-limit enforcement. The gateway deliberately does not know anything about business logic — it forwards requests to one of the downstream service clusters based on a routing table generated by the platform team.

The **core services layer** holds the domain logic. Services are grouped as `shipments`, `carriers`, `depots`, `pricing`, and `notifications`. Each service is a Go binary with its own Postgres schema. Cross-service calls travel on gRPC with protobuf definitions generated from a central `proto/` directory. Services discover each other through a Consul catalog; service addresses are refreshed every thirty seconds.

The **worker layer** consumes asynchronous jobs from Kafka topics and NATS subjects. Workers handle long-running operations that should not block HTTP responses: carrier rate-card refresh, bulk shipment imports, customs document generation, and webhook delivery. Worker pools autoscale based on topic lag; each worker type has its own scale policy so that a spike in one stream does not starve another.

The **batch layer** sits off to the side as a Python-based ETL that runs nightly. It consolidates shipment events into an analytical warehouse and produces rollup reports. Batch jobs are orchestrated by Airflow and can be replayed from any point in the last 90 days. The batch layer has read-only access to the core service databases through dedicated replica credentials.

## Domain Terminology

These terms recur throughout the codebase, the database schemas, and the API reference. They often differ from industry conventions because the original team adopted local jargon early on.

A **shipment** is a single parcel tracked end-to-end. Each shipment owns a unique identifier (`SHP-...`), a status history, a chain of custody events, and a billing record. A shipment can have multiple legs — a leg is one segment of transport between two handoff points. The legs together form the shipment's route.

A **depot** is a physical facility where parcels are sorted, consolidated, or cross-docked. Depots have capacity limits, operating hours, and a hierarchy (regional hub, local station, last-mile outpost). The depot subsystem also tracks yard assets such as trailers and intermodal containers awaiting departure.

A **carrier** is any external logistics provider integrated with Nimbus. Carriers register through the onboarding portal, upload proof of insurance, and submit a rate card per lane. A lane is defined as a (origin region, destination region, service level) triple. Rate cards are versioned — older rates remain available for historical invoices.

A **handoff** is the event in which custody of a parcel transfers between two parties. Handoffs can happen between carriers, between a carrier and a depot, between a depot and a courier, or between a courier and the recipient. Every handoff must resolve to either a successful transfer or an exception with a reason code from the shared enum.

A **service level** (SL) describes the commitment attached to a shipment: same-day, next-day, economy, or bulk. Service levels influence routing, pricing, and the exception handling rules. The SL field is immutable after shipment creation — a customer who needs a different SL must cancel and recreate the shipment.

A **customs profile** is the combination of HS codes, declared values, currency, importer identity, and shipper identity associated with a shipment that crosses an international border. Customs profiles are versioned because regulators occasionally ask for retroactive amendments.

## Data Model Highlights

The data model is shaped by five core tables that every service team eventually touches.

`shipments` is the widest and most queried table. Columns include `id`, `created_at`, `origin_address`, `destination_address`, `service_level`, `status`, `status_updated_at`, `external_reference`, `customs_profile_id`, and a JSONB `metadata` blob used for carrier-specific fields. The table carries partial indexes on `status` filtered to active shipments for fast dashboard queries. Writes go to the primary; reads split between primary (for consistency-critical paths) and two streaming replicas.

`legs` stores the per-segment transport records. A leg has a `shipment_id`, an `origin_depot_id`, a `destination_depot_id`, a `carrier_id`, an `expected_departure`, an `actual_departure`, an `expected_arrival`, an `actual_arrival`, and a `status`. Legs are created by the routing service when a shipment is planned; they are mutated by tracking events and by manual overrides from depot operators.

`events` is an append-only log of domain events. Columns: `id`, `shipment_id`, `leg_id` (nullable), `source`, `type`, `payload` (JSONB), `recorded_at`, `ingested_at`. Events come from many places: carrier webhooks, scanner devices at depots, operator console actions, and reconciliation jobs. Because the stream is append-only, a common pattern is to materialize derived views rather than mutate rows.

`rate_cards` holds pricing data. Columns: `carrier_id`, `lane_id`, `effective_from`, `effective_until`, `base_price`, `currency`, `surcharge_rules` (JSONB). Only one row per `(carrier, lane)` is considered active at any moment; overlapping intervals are rejected at the database level by an exclusion constraint.

`users` is the multi-tenant user directory. Each user belongs to exactly one tenant and zero-or-more roles within that tenant. Roles map to permission bundles defined in the authorization service. Password hashes have not been stored in this table since 2022 — authentication delegates to an external identity provider.

## Entity Relationships

Shipments own legs; legs reference depots and carriers; depots belong to a network; carriers own rate cards. Events attach to shipments (always) and optionally to legs. Rate cards are independent of individual shipments but are snapshotted onto the shipment's billing record when priced.

The relationship between tenants and shipments is one-to-many — each shipment is owned by exactly one tenant. Cross-tenant visibility exists only for reconciliation purposes and is mediated by a dedicated audit role.

The authorization graph is represented as `(user, role, scope)` triples where scope can be a tenant, a depot, or a carrier. Role definitions live in a separate config repository and are loaded on service boot.

## API Surface

The public API exposes four top-level resource roots: `/shipments`, `/carriers`, `/depots`, and `/tenants`. Each root has the usual CRUD verbs plus a set of domain-specific sub-actions.

On `/shipments`, notable sub-actions include `POST /shipments/:id/transitions` (state machine moves), `POST /shipments/:id/holds` (temporary freeze for customs), `POST /shipments/:id/exceptions` (record a delivery exception), and `GET /shipments/:id/timeline` (chronological event list). The timeline endpoint is heavily used by customer-service dashboards and by the carrier-facing mobile app.

Pagination uses cursor tokens rather than offsets. Cursors are opaque base64 strings containing the primary key and a timestamp; they cannot be parsed by clients. Server-side, cursors map to a keyset index scan on the filtered table. Page sizes are capped at 500.

Filtering uses a subset of the RSQL query language, parsed by a shared middleware. Clients can pass complex expressions like `status=in=(in_transit,delayed);service_level==express`. The middleware rewrites RSQL into parameterized SQL conditions before reaching the services.

Response envelopes always carry a `meta` field with `request_id`, `trace_id`, `timestamp`, and a `links` collection for HATEOAS-style navigation. Clients that care about performance often ignore the links and navigate by hardcoded URLs.

## Asynchronous Workflows

Several flows span minutes or hours, making them unsuitable for the synchronous request-response model.

**Bulk import** accepts a CSV or JSON file up to 50MB. The import service validates the file header, splits it into batches of 1,000 rows, and publishes each batch to the `imports.batch` Kafka topic. Worker instances pick up batches, perform row-level validation, and either create shipments or write failures to a dead-letter topic. The import report aggregates successes and failures and emails the requesting user.

**Carrier rate-card refresh** pulls updated pricing from carrier portals nightly. The refresh worker uses a combination of SFTP polling (for legacy carriers) and REST pulls (for modern ones). Each refresh generates a diff report against the previous rate card. Large diffs (more than 20% of rows changed) go to a human reviewer before activation.

**Customs document generation** is triggered when a shipment crosses an international border. The worker gathers the customs profile, HS codes, commercial invoice line items, and any required certificates, then composes a PDF and a structured CUSDEC message. Both artifacts are uploaded to the storage bucket and linked to the shipment record.

**Webhook delivery** pushes shipment status changes to tenant-owned URLs. Deliveries are retried with exponential backoff up to six attempts. Persistent failures are recorded and surfaced in the tenant's admin dashboard for manual replay.

**Reconciliation** runs every four hours. It compares carrier-reported scans with internally recorded handoffs and flags mismatches. Flagged items enter a review queue worked by an operations team across time zones.

## Event System

Events are produced by many sources and consumed by many services. To keep the graph tractable, every event type is documented in `events.yaml` with a schema, a producer list, a consumer list, and a retention policy.

Kafka is used for high-throughput event streams where ordering per partition matters. Partition keys are almost always the shipment ID, so all events for a single shipment land on the same partition. NATS is used for lower-volume control-plane messages where at-most-once delivery is acceptable.

A small internal event bus (homegrown, built on Redis streams) handles intra-service events that never need to cross process boundaries. It exists because it predates the Kafka rollout and teams have not migrated the last few call sites.

## Caching Strategy

The Redis keyspace is partitioned by service. Each service has a prefix (`shp:`, `crr:`, `dep:`, `pri:`, `ntf:`) and is allowed to write only to its own prefix.

The shipment service caches the latest status plus the last five events for each active shipment. TTL is 30 minutes, refreshed on every event. Cache misses fall through to Postgres.

The pricing service caches computed quotes for popular lanes. Cache keys incorporate the carrier ID, the lane triple, the service level, and the weight bucket. TTL is 5 minutes. The cache is explicitly invalidated when a new rate card becomes active.

The depot service caches the current capacity view, refreshed every 30 seconds. Capacity is a derived number based on yard occupancy plus pending arrivals; it is expensive to compute on demand.

Invalidation uses a publish-subscribe channel (`cache:invalidate:<prefix>`) that each service subscribes to. Writers publish on the channel after a mutation that should purge cached data.

## Observability

Three telemetry streams flow out of the platform: logs, metrics, and traces.

**Logs** are structured JSON, emitted to stdout, and collected by the Vector pipeline. Vector forwards logs to Loki for long-term storage and to a BigQuery sink for ad-hoc SQL analysis. Every log line carries a correlation ID that matches the `request_id` header on inbound traffic.

**Metrics** use Prometheus with a scrape interval of 15 seconds. Each service exposes a `/metrics` endpoint behind a network policy that restricts access to the monitoring subnet. Metrics are federated across regions through a Thanos hierarchy.

**Traces** use OpenTelemetry exporters configured to send to a Tempo cluster. Sampling is head-based at 5% for normal traffic and 100% for requests flagged as debug by internal operators. Trace context propagates through HTTP, gRPC, Kafka, and NATS via W3C Trace Context headers.

Alerts are defined in Alertmanager and routed by severity and ownership. Critical alerts page the on-call via PagerDuty; warning alerts fall into a Slack channel watched during business hours.

## Deployment Topology

Production runs across three cloud regions: `apac-east-1`, `apac-south-1`, and `apac-west-1`. Each region hosts a full service stack capable of serving live traffic; cross-region replication is handled by each service's own database layer (Postgres logical replication for core tables, Debezium feeds for event fan-out).

Deployment is orchestrated by Argo CD from a central GitOps repository. Merges to `main` generate new Helm chart versions, which are applied to the dev cluster automatically, to the staging cluster after a successful integration suite, and to production after a human approval.

Feature flags are managed through Unleash. Flag changes take effect within 60 seconds of being saved in the Unleash UI; services poll the API for updates. Some flags are scoped per-tenant to allow gradual rollouts to specific customers.

Config data beyond feature flags lives in HashiCorp Vault, accessed through a per-pod sidecar that injects environment variables at startup. Secret rotation is tracked quarterly, with each service responsible for accepting the new secret before the old one expires.

## Security Model

Authentication and authorization are separated. Authentication happens at the gateway — JWTs are issued by a dedicated identity provider and contain subject, issuer, audience, and expiration. The gateway verifies the JWT signature and sets an internal header (`X-User-Id`) that downstream services trust.

Authorization happens in each service. Services load a permission bundle for the user and check it against the resource the request targets. The authorization layer uses a policy-as-code engine with policies stored in Rego files, evaluated through OPA.

Data at rest is encrypted via cloud-provider KMS for object storage and Postgres TDE for relational data. Backups are encrypted with a separate key and stored in a different region.

Network policies restrict service-to-service traffic to a minimal set of allowed pairs. Internet-facing ingress is limited to the gateway; no backend service has a public endpoint.

Audit logs capture authentication events, permission changes, and destructive mutations (deletes, bulk updates). Audit log retention is seven years to match compliance requirements.

## Migration History

The codebase has seen three major migrations that still influence how things look today.

**The 2020 monolith split** broke the original single-binary dispatcher into the five services in use today. Traces of the monolith remain as a `legacy-core/` directory whose purpose is historical reference — nothing in that directory is loaded at runtime, but some module names were preserved when ports were split out.

**The 2022 identity migration** moved user records out of the core database into an external identity provider. The migration left behind a set of shim columns (`legacy_user_id`, `legacy_tenant_id`) that are still referenced by analytical jobs. A stated goal of the platform team is to remove the shims once the warehouse is updated, which has been on the roadmap for two years.

**The 2024 event streaming migration** moved intra-service event delivery from Redis streams to Kafka for most topics. A handful of low-volume topics remain on Redis streams because the migration cost outweighs the benefit for those flows.

## Historical Decisions

Every major technical decision since 2021 has been captured in the `docs/adr/` directory using MADR templates. Selected highlights are reproduced here in summary form to illustrate how the team thinks about trade-offs.

ADR-017 "Postgres partitioning for the events table" chose time-based range partitioning over list partitioning because the query patterns are strongly time-oriented. The decision accepted a higher maintenance overhead in exchange for simpler query plans.

ADR-024 "Go as the primary service language" cemented the team's move from a mixed Node/Python stack to Go for new services. The main reasons cited were predictable performance, deployment simplicity, and a shared skill base across the team. Existing Node services were allowed to remain but are not expected to receive new features.

ADR-031 "gRPC for internal RPC" chose gRPC over a REST-based internal API to obtain strong typing and bidirectional streaming. A REST façade still exists at the gateway; internal calls from there to downstream services are translated to gRPC.

ADR-038 "Polymorphic message envelope for webhooks" introduced a single envelope shape with a discriminant field. Earlier webhooks had inconsistent shapes between services and were painful to consume.

ADR-044 "Default Postgres isolation level" set `READ COMMITTED` as the default for all services, with explicit opt-in to `REPEATABLE READ` for a handful of jobs that need stronger guarantees. This choice came after a week-long incident caused by unexpected serialization failures under heavier concurrency.

ADR-051 "Airflow for batch orchestration" chose Airflow over dbt Cloud because the existing team already operated an Airflow cluster for data engineering. Switching would have doubled the tooling footprint without a clear payback.

## External Integrations

The platform connects to a long list of external services. Each integration has a dedicated module with its own retry and error-handling policy.

**Carrier portals** — each major carrier provides a different API surface, ranging from modern REST to SOAP with XML payloads. The `carrier-integrations/` directory contains one adapter per carrier, each implementing the same `Carrier` interface. Adapters handle authentication, request shaping, rate limiting, and response parsing.

**Customs systems** — customs authorities in each jurisdiction expose different interfaces. The European Union uses ICS2. Several APAC jurisdictions use private EDI connections. North America uses ACE. Each adapter normalizes the response into the same internal structure.

**Payment providers** — billing flows through Stripe for card payments and a regional bank gateway for direct debits. Invoice generation uses an external document service that renders PDFs from Handlebars templates.

**Map and geocoding services** — Mapbox is the default provider; OpenStreetMap is used as a fallback for addresses in regions where Mapbox coverage is poor.

**Notification channels** — email goes through SES, SMS through Twilio, and push notifications through Firebase. Each channel is backed by a preference record per recipient that determines whether a particular event type should be delivered.

## Background Jobs

Beyond the worker pools, a set of smaller background jobs run on cron schedules.

`reaper` runs hourly and closes shipments that have been in a terminal state for more than 24 hours. Closing means moving the shipment to the archive table and removing the cached row from Redis.

`catchup` runs every 15 minutes and retries webhook deliveries that failed on their initial attempt. It does not replace the main retry loop; it exists to catch webhooks that slipped through due to transient broker issues.

`freshness` runs every 10 minutes and refreshes the materialized view used by the depot capacity dashboard. The view is not automatically refreshed by Postgres because the refresh is expensive relative to the value.

`pricing-snapshot` runs every 30 minutes and computes an aggregate view of active rate cards for the business intelligence team. The aggregate is written to a separate schema that BI tools can query without hitting the operational tables.

`pii-scrubber` runs nightly and redacts PII fields on events older than the configured retention window. The original row remains for audit, but the fields are overwritten with a hash.

## Rate Limiting

Rate limits are enforced in two places.

The gateway applies coarse limits per API key and per IP address. These limits use a token-bucket algorithm backed by Redis. Exceeded limits return HTTP 429 with a `Retry-After` header.

Individual services apply fine-grained limits per operation when the gateway cannot distinguish between cheap and expensive calls. The pricing service, for example, has a per-tenant limit on quote calculations to prevent accidental DoS from a broken client.

Rate-limit breaches are logged and surfaced in a dashboard that tenants can view in their admin UI. Repeated breaches over a 7-day window trigger an outreach from the customer success team.

## Internationalization

Nimbus operates across ten countries and supports eight interface languages. Language is selected per user via a profile setting; the default language for a new user is inherited from the tenant.

Text strings live in Crowdin. The build pipeline pulls the latest translations every night and produces language bundles that are shipped with the frontend. Backend error messages are translated at the edge of the gateway — internally, services emit machine-readable error codes and the gateway looks up the localized string.

Currency display uses the tenant's billing currency by default but can be overridden per user. Currency conversion rates come from a daily snapshot fetched from an external FX provider; the snapshot is timestamped and stored alongside billing records for auditability.

Date and time display honors the user's time zone where known and falls back to the tenant's time zone otherwise. Internally, timestamps are stored as UTC without exception.

## Developer Onboarding

New engineers on the team typically spend their first week reading documentation and pairing with an onboarding buddy. The onboarding plan is maintained in the handbook and revised every quarter.

The local dev environment uses Docker Compose to bring up Postgres, Redis, Kafka, NATS, and a minimal set of services. The services run in hot-reload mode, backed by a shared volume. Boot time from a cold checkout is typically under three minutes.

Tests run through a shared script that selects the right subset based on the directories changed in a pull request. The full suite takes about eight minutes on the CI cluster. Coverage has held at 78% for the core services for the last year.

Code reviews are required on all pull requests and typically have at least one reviewer from another team. Reviewers look for correctness, test coverage, and alignment with the domain model. Style is enforced by automated formatters, so reviewers rarely discuss it.

## Release Cadence

The platform follows a weekly release train. Changes merged to `main` by Friday afternoon are included in the following Tuesday's deployment. Hotfixes can go out at any time through a separate expedited pipeline.

Each release is accompanied by release notes drafted from conventional commit messages and reviewed by a product manager. Notes are published to a shared Confluence space and sent to a Slack channel watched by the wider organization.

Rollbacks are possible on any release through the GitOps repository. A rollback re-applies the previous chart version and reverts any database migrations that were marked rollback-safe. Migrations marked irreversible require an incident response.

## Known Trade-offs

The team consciously accepts several trade-offs to keep the platform sustainable.

Query latency for historical shipments (older than 90 days) is higher than for recent shipments because historical queries hit the cold storage instead of Redis. The UI communicates this by showing a "loading" indicator for archived lookups.

The polyglot storage layer (Postgres, Redis, object storage, Kafka, Airflow database) makes local development heavier than a single-store alternative. The benefit is that each service can pick the storage that fits its access pattern.

Cross-region failover is currently manual. An automated failover was considered in 2023 but deferred because the team was unsure the additional complexity would pay off given how rarely regions fail.

The public API supports both the v1 and v2 shapes. v1 was frozen in 2022 but still carries about 15% of traffic from long-lived integrations. Retiring v1 has been on the roadmap but is constantly deprioritized in favor of new work.

## Retired Subsystems

Several subsystems that used to exist have been retired. Their residue remains in the code for historical reasons.

The **legacy dispatcher** was the original service that handled shipment routing. It was replaced in 2021 by the current routing service, which has a very different design. Two small modules from the legacy dispatcher were extracted into a utility library because other places depended on them.

The **email-only notification engine** handled tenant communication before the multi-channel notifications service was built. Its templates live in a dormant directory and are never loaded at runtime.

The **in-house feature flag system** was replaced by Unleash in 2023. Some services still import a compatibility shim that routes flag lookups to Unleash under the old API.

## Third-Party Vendors

The team tracks every third-party vendor in a central spreadsheet that is reviewed quarterly. Each row records the vendor name, the service it provides, the owning team, the contract expiry, the security review status, and the estimated annual spend.

Vendors considered critical (production-impacting if unavailable) have documented failover plans. Non-critical vendors have a lower bar.

## Reference Datasets

A handful of reference datasets are loaded into the platform and rarely changed.

`countries.yaml` holds ISO country codes, default currencies, default tax rates, and customs flags.

`postal_codes/` holds country-specific postal code validators and geocoding helpers. Each file is maintained by a volunteer from the regional team that understands the local addressing conventions.

`hs_codes.csv` is a snapshot of the harmonized system code tree used for customs declarations. The snapshot is refreshed once a year after the WCO publishes updates.

`holiday_calendar.yaml` lists non-business days per country. Delivery SLAs account for these dates.

## Historical Incidents

The team publishes post-mortems for incidents rated S1 and S2 in an internal knowledge base. The post-mortems are written in a no-blame format and focus on contributing factors and remediation items.

Incident 2022-03-14: a rate-card refresh failure put the pricing service into a degraded mode for six hours. Root cause was a missing null check in the carrier adapter. Remediation introduced a pre-activation validation step.

Incident 2023-08-02: a Kafka rebalance storm caused worker pools to fall behind by several hours. Root cause was an excessive number of consumer group members for a partition count that had not grown with traffic. Remediation rebalanced partitions and introduced autoscaling limits.

Incident 2024-11-19: a bad Helm chart template deployed a service with zero replicas. Root cause was a newly-introduced conditional in the values file. Remediation added a CI check that catches zero-replica deployments before they are applied.

## Notes on Documentation Style

The handbook recommends a descriptive style for long-form documents like this one. Sentences describe what is, not what should be. When a rule or preference exists, it is worded as an observation of existing practice rather than as an instruction to the reader. This style keeps the document readable alongside generated artifacts and prevents it from being mistaken for runnable policy.

## Reading the Codebase

The repository is organized so that each service owns a top-level directory under `services/`. Shared libraries live under `libs/`. Infrastructure definitions live under `deploy/`. Protocol definitions live under `proto/`. Documentation lives under `docs/`.

A fresh clone takes about 4GB of disk once dependencies are installed. The dependency graph is flat enough that most files can be opened and understood without first walking through a package tree, provided the reader is familiar with Go project conventions.
